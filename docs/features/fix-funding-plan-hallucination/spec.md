# Specification: Fix Funding Plan Hallucination & Remove Hardcoding

## 1. 개요 (Overview)
현재 부동산 리포트 생성 프롬프트(`insight_parser.md`) 내에는 두 가지 핵심적인 문제 상황이 존재합니다. 본 문서는 이를 해결하기 위한 개선 계획을 수립합니다.

1. **LLM의 '자금조달계획' 산술식 오류 (Hallucination)**
    - LTV 최대 한도를 고려하지 않고, `보유 자본금 + DSR에 따른 대출 한도`를 그대로 덧셈하여 현실적으로 비가능한 매수 예산을 제시함.
2. **프롬프트 내 지표들의 과도한 하드코딩 (Hardcoding)**
    - 페르소나 (예: 자본금 3억, 부부 합산 소득 1.6억, 삼성/선릉역 등)
    - 연도 및 금융 환경 (예: 2026년, 스트레스 DSR 3단계)

## 2. 문제 원인 분석 (As-Is)
- 프롬프트 4번 "나를 위한 맞춤형 액션 플랜"에서 페르소나의 세부 스펙(소득, 예산, 선호 지역)과 금융 규제 단계가 통째로 정적으로 박제되어 있습니다.
- 이로 인해 향후 사용자의 연봉 증가 ‧ 이사 목표 변경, 혹은 2027년 및 새로운 금융 규제가 도래할 경우 매번 프롬프트 파일(마크다운) 코드를 뜯어고쳐야 하는 유지보수 이슈가 생깁니다.
- 또한 산술적 오류 제약 조건이 누락되어 있어 LTV 역산액(현금 기반)보다 초과된 값으로 예산안을 도출하게 둡니다.

## 3. 개선 방안 (To-Be)

코드는 수정하지 않고 오직 **프롬프트 문서의 지시 내용**만을 이하 방향으로 편집하여 동적 의존성 주입을 활용하도록 만듭니다. 아울러 핵심 서비스 로직(`service.py`)에 자가 검증 루프와 웹 검색 엔진 연동을 추가합니다.

### 3-1. 하드코딩 제거 및 동적 변수화
- **자본금/소득/선호지:** "사용자 자본금(3억)... 삼성역/선릉역 직주근접과..." 대신 **"입력된 사용자 페르소나 값에서 추출한 총 가용 자산, 현재 소득, 직장 위치 및 선호 거주지 등의 변수를 동적으로 참조하여 판단할 것"**으로 교체.
- **연도/시장 상황표준:** "2026년 현재"라는 고정 단어를 제거하고 **"입력 데이터에서 제공된 `target_date` 시점을 기준으로"** 작동되도록 교체.

### 3-2. LTV/DSR 교차 검증 논리식 삽입 
매수 예산 산출 로직에 아래 **제약 지시어**를 강제합니다.
1. 절대 `(페르소나의 가용현금 + 산출된 DSR 대출한도액)`으로 단순 합산하여 최대 예산을 도출하지 말 것.
2. 먼저 규제 정책에 명시된 LTV 비율에 근거하여, 가진 현금만으로 매입 가능한 주택 실거래가 상한선**(LTV 역산 한도 = 가용 현금 / 1 - LTV 최고치)**을 수학적으로 계산할 것.
3. 이를 DSR 한도 대출금액과 결합했을 때의 한도와 비교하여, 더 보수적인 값을 '최종 매수 가능 예산'으로 도출할 것.

### 3-3. 실시간 웹 검색 기반 금융 정책 컨텍스트(Policy Context) 수집
- 하드코딩된 정책(예: 스트레스 DSR 3단계) 대신, 파이썬 백엔드에서 리포트 생성 시점에 **웹 검색(DuckDuckGo Search 등)** 또는 **실시간 검색 가능 LLM**을 활용해 "최신 주택담보대출 LTV, DSR 정책"을 검색하고 그 결과를 `policy_context`에 동적으로 반영합니다.

### 3-4. 논리적 타당성 자체 검증 루프(Self-Reflection Loop) 도입
- 생성된 액션 플랜이 수학적으로 불가능하거나 비합리적인 제안(예: DSR 초과, 터무니없는 예산 등)을 담고 있지 않은지 LLM 스스로 평가하는 단계를 추가합니다.
- 평가 기준 10점 만점 중 **8점 이하**일 경우, 이전 결과의 논리적 오류를 지적하는 프롬프트와 함께 리포트 재생성을 지시합니다.
- 무한 루프를 방지하기 위해 **최대 10회**로 반복을 제한합니다. 만약 10회 시도 후에도 8점을 넘지 못하면, 리포트 결과 하단에 *"⚠️ 본 액션플랜은 AI 논리적 타당성 검증에서 낮은 점수를 받았으므로, 주거래 은행과의 상세한 상담을 필히 권장합니다."*라는 경고성 명시 문구를 노출하도록 설계합니다.

## 4. 진행 순서
1. 사용자가 본 수정 기획안(`spec.md`)의 방향성을 컨펌합니다.
2. 봇이 `src/modules/real_estate/prompts/insight_parser.md` 파일 내의 하드코딩된 값 지우기 및 규칙 추가(Execution)를 수행합니다.
3. 리포트를 발송/생성하여 2027년이나 다른 조건이 투입되더라도 유연하게 작동하는지 & 10억 언저리의 합리적인 예산을 뽑는지 최종 검증(Verification)합니다.
